$\text{no}\TeX\text{book}$ shall be thy name¶

When I designed this theme, I was aiming towards a notebook experience which could bring the elegance and sobriety of a $\LaTeX$-generated article, into the flexibility and interactivity of Jupyter. The name of this project had to immediately convey this idea of integration. def hex2rgb(h: str):
    """helper to convert hex colours into RGB tuple"""
    h = h.lstrip('#')
    return tuple(int(h[i:i+2], 16) for i in (0, 2, 4))

# Create a black image
image = np.full((1024,1024,3), 255, np.uint8)

def show_palette(colours):
    n_cols = len(colours)
    fig = plt.figure()
    for i, (colour, hexcode) in enumerate(sorted(colours.items(), key=lambda e: hex2rgb(e[1]))):
        axis = fig.add_subplot(1,n_cols,i+1)
        rgb = hex2rgb(hexcode)
        cv2.circle(image, (512, 512), 512, (rgb[0], rgb[1], rgb[2]), -1)
        plt.title(colour, y=-0.5)
        plt.axis('off')
        plt.imshow(image);

Fonts and Typeface¶

All the fonts used in the $\text{no}\TeX\text{book}$ theme typeface design are open source, and freely available on public font repositories (e.g., Google Fonts, CDN Fonts, FontSquirrel). All the main fonts used are released under the OFL (Open Font License), and alike. Original reference and download links are provided below $\Downarrow$.

$\text{no}\TeX\text{book}$ typeface¶

The main goal of the $\text{no}\TeX\text{book}$ theme is to allow Jupyter notebooks to render like a $\LaTeX$-generated article. Therefore, the (Markdown) text rendering adopts the good ol' Computer Modern font family for both serif and monospace text (i.e. CMUSerif and CMUTypewriter fonts, respectively).

In edit mode, the Hack font is used for Markdown, whereas Fira Code font is used for code editor. The former looks slightly better for text writing (imho), being also based on the Bitstream Vera and the DejaVu fonts; whereas the latter has been specifically designed for source code, combining an elegant style with support for ligratures$^1$.

[1]:The support for ligatures in the Markdown editor would be almost useless anyway, being then rendered in Computer Modern font.

Notebook UI¶

The font used for everything else in the Notebook UI and look-and-feel (e.g. header, navbar, jupyter-server home page (notebook list) uses the Roboto Slab font.

Footnotes¶

To define a new footnote which complies to the CSS directives specified in the $\text{no}\TeX\text{book}$ theme, the following HTML code should be used:

<span id="fn1-ex"><i>[1]:</i> This is an example of a footnote</span>

Footnotes could be identified by an id starting with the fn prefix (the CSS matching rule is [id^='fn']). The definition of footnotes via (unique) ids will also make footnotes to be automatically retrieved via anchor links in footnote reference.

It is also possible to emphasize the footnote reference using the <i> tag.

The resulting style of a footnote$^1$ is as follows:

[1]: This is an example of a footnote

If we want to stack multiple$^{1st}$ footnotes in the same Markdown cell$^{2nd}$, the CSS will comply:

[1]: This is an example of a footnote [2]: This is the **second** example of a footnote

Last but not least, if we want or need to identify a footnote with a specific id that will not comply with the fn prefix requirement, the .fn class can be used instead$^{CSS}$:

<span id="css" class="fn"><i>[1]:</i> This is footnote of class `.fn` and any `id`</span>

[CSS Class] `.fn` : This is footnote of class `.fn` and any `id`

Image Width¶

As default, the max-width of images (and SVG) is automatically capped at the 70% to allow for a proper rendering of large images, avoiding any potential overflow. However, there may be cases in which it could be required to control the max-width parameter in order to enlarge or shrink an image. To do so, some special CSS classes have been defined to control the value of max-width attribute.

Code Editor¶

The CSS style for code and markdown editor integrated in $\text{no}\TeX\text{book}$ adopts a strategy that aims at flexibility and extensibility.

The main style file for code editors is editors.css. This file imports each theme for both code and markdown editors. The themes for the two editors are defined in two different files, to allow for maxmimum flexibility in the choice of the colours schemes to adopt.

In each theme file, the base colour palette is defined, along with the mapping of each of these colours to specific (editor) variables. These variables are then used to define the CSS style rules.

In this way, it is very easy to change or try multiple themes, or even defining custom new ones.

The default colour scheme used for code and markdown editor is based on the Material Design Light Theme. In addition, other editor themes are bundled with the $\text{no}\TeX\text{book}$ Jupyter theme, e.g. GitHub Light for code, Typora theme for markdown.

The base colour palettes for each of those themes are showed below.

Google Colaboratory Support¶

The HTML of a Colaboratory notebook is nothing similar to the standard Jupyter notebook/lab HTML structure (so the custom-css cannot work). In addition, every single Colaboratory notebook cell generates a new HTML iframe, so making the texify IPython magic pretty much useless too.

Therefore, the only solution I could resort to was using the Stylus) browser extension.

Unfortunately it is not as customisable and portable as the IPython magic is, but hey?! Better than nothing 🤓

If you wanna have a peek, let's follow all the instructions ( here $\Rightarrow$) on how to install $\text{no}\TeX\text{book}$ for Google Colaboratory Notebooks.

Once you are done, try to open the sample notebook in Colab, and enjoy 🙌
